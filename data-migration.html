Technical Investigation · Rev. 2

Where Winds Meet data: source map & the Route B build plan

Where wwmdb.vlt.fyi gets its data, and a concrete plan to build gamevn-wwm on first-party sources — covering static game data and live player, guild, and ranking data, with the exact APIs.

Prepared for Khanh Subject: wwmdb.vlt.fyi · sixhorse.game.163.com 2026-07-19

The short version

wwmdb is a thin wrapper over two upstreams: it datamines the game client for static data (items, gear, recipes), and it proxies NetEase's official game API (sixhorse.game.163.com) for live data — player profiles, gear loadouts, guilds, and leaderboards. Its own /api just re-serves both behind a token hardcoded in its public JavaScript.

So Route B splits cleanly in two: get static data from the open ejtechdev/wwm-data repo (or datamine the Messiah-engine client yourself), and get live data from the official NetEase API the way the official Data Tool does.

Recommendation: stand up the static layer first (it's easy and clean), then add the live layer against the official API — and read the privacy caveat before you store a single player record.

01 What the site actually is

A fan-built Where Winds Meet database (author handle zelenin, with Discord and Buy-Me-a-Coffee links). The front end is a SolidJS single-page app; the bundle carries no data and fetches every record at runtime from a backend on the same domain.

02 Where the data comes from

Their /api is a facade. Behind it sit two genuinely different upstreams — and that split is the key to your own build.

  • Upstream A · Static game data

    Datamined from the game client

    Items, gear, affixes, crafting recipes, inner ways, suits, books, world levels. Records carry the game's internal IDs and tables (e.g. item 1100001 "True Heart - Sword"). Extracted from the client's own config assets.

    Note: the game runs on NetEase's proprietary Messiah engine, not Unreal — so the usual Unreal toolchain (FModel, UnrealPak) does not apply. This makes a from-scratch datamine non-trivial.

  • Upstream B · Live account data

    Proxied from NetEase's official API

    Player profiles, gear loadouts, guild membership, and leaderboards are pulled live from sixhorse.game.163.com — the same NetEase backend that powers the official Data Tool. Requests key on roleId + hostnum (a region/server number) and return current, per-account state.

    I confirmed this end-to-end: a single lookup returned a player's level, weapon mastery, region, gear — and even their NetEase account email. wwmdb is effectively a cache/relay in front of this.

03 wwmdb's own API, concretely

Every call is POST /api/wwm.v1.WwmService/<Method> (ConnectRPC/JSON), with an X-Language header for locale. The "auth" is client-side obfuscation over a token copied verbatim from the public bundle:

// hardcoded in the site's own JavaScript
const token = "ab964c45612bda768691108730d0c31c9b77116449fc99c3d4dc29d17db2cd77";
nonce   = crypto.randomUUID();
payload = `${token}:${unixSeconds}`;
bearer  = base64(xor(payload, nonce));   // repeating-key XOR
// headers →  Authorization: Bearer ${bearer}   X-Request-Id: ${nonce}

Static methods (EquipItems, InnerWays, …) take {}. Live methods take keys: SearchUser {"search":"…"}, Player {"id":"<oid>","hostnum":<n>}, Guild {"id":…}, Rank {"id":…}. This is Route A, and it works — but it's their private relay, not yours.

◆ 4.1 · Static game data

This layer is nearly free. Two options, in order of effort:

Primary — recommended

ejtechdev/wwm-data

Open community repo of WWM game data (JSON) and icons, updated by PRs. Pull it as raw files or vendor it into your build; no auth, no scraping.

raw.githubusercontent.com/ejtechdev/wwm-data

Fallback — for gaps

Datamine the client

For entities the repo lacks, extract from the Messiah-engine client yourself using NetEase/NeoX-family unpackers. Higher effort; proprietary format, no turnkey tool.

Re-run per game patch

Coverage is the same catalogue wwmdb exposes: gear & affixes, stuff/guise items, compound recipes, inner ways, suits, books, world levels, attribute translations, riddles.

◆ 4.2 · Live data — the official NetEase API

This is the real work, and the only source of player/guild/ranking data. The backend is sixhorse.game.163.com (mirror: sixhorse.game.easebar.com), fronted for the web by the official Data Tool. Access model:

ConcernHow it works
Base hostsixhorse.game.163.com — NetEase's "Six Horse" game-data service
AuthNetEase MPay / UniSDK oversea login (the Data Tool loads mpay-h5-sdk-oversea). You log in with a real NetEase account and get a session bound to it.
KeysroleId (the player's oid) + hostnum (region/server) + region/oversea tag
RegionGlobal/oversea servers are a separate cluster from CN — match the tool your players use (this build targets oversea)

The Data Tool organizes it into planes that map directly onto what you asked for:

You wantData Tool planeWhat you get
Player gearCombat Stats / MineProfile, level, weapon mastery, equipped gear + affixes, inner ways, mystic skills
Guild & rankingRatingsGuild membership + leaderboards (e.g. sect election polls, healer rankings, speedrun boards)
Match infoRatings / RankCompetitive boards like Tempest Speedrun Qualifier; see the honesty note below on true per-match history

Honesty note on "match info"

The official surface exposes leaderboards and rating snapshots, not a per-match replay/history feed. wwmdb's Rank/RankGroups return board standings (Tempest Speedrun, election polls, etc.), and Player returns current loadout — but I found no evidence of a match-by-match history endpoint. If detailed match logs exist, they'd need to be discovered on sixhorse.game.163.com directly; assume for now that "match data" means ranked leaderboards + current standings.

◆ 4.3 · Your stack

Ingestion

Two sync cadences

Static: re-sync from wwm-data on each game patch (rare). Live: fetch on-demand per lookup, then cache with a short TTL — never hammer the official API per page view.

Serving

Your own API + store

Normalize both planes into your DB, host icons on your own CDN, and expose your REST/GraphQL to the gamevn-wwm frontend. Your app never talks to an upstream directly at request time.

◆ 4.4 · Suggested sequence

  1. Static layer — low risk

    Vendor ejtechdev/wwm-data, normalize into your schema, self-host icons, ship a read-only game database. This alone is a usable product.

  2. Live read path — prototype via Route A

    To validate UI shape fast, read live data through wwmdb's API while you build. Cache everything you fetch into your own store from day one.

  3. Official API integration — the durable path

    Reverse the Data Tool's calls to sixhorse.game.163.com, implement NetEase MPay login, and replace the Route-A calls with first-party ones.

  4. Harden — before public launch

    Rate-limit, cache TTLs, PII scrubbing (below), attribution, and a patch-day re-sync job.

Read before storing any player data

  • PII leak: the live API returns players' NetEase account emails. Do not store or display them — strip account/email fields at ingestion. Publishing them would be a serious privacy problem.
  • Terms & auth: the official API is NetEase's and gated behind their login/ToS. Automated access at scale may violate it — keep call volume modest, cache hard, and be ready to seek permission for anything public and commercial.
  • IP: game data and assets remain Everstone/NetEase/Tencent IP. Attribute clearly; keep it non-commercial unless cleared.

My call: ship the static layer on Route B now, prototype the live layer through Route A, and migrate the live layer onto the official API before anything is public — with PII stripped and attribution in place.
